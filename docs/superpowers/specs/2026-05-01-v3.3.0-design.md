## 三、详细设计

### 3.1 #15 — COMPLETE_PATCH 增加平台判断

**根因**：`_render_complete_hook_block()` 生成的注入代码在所有平台触发 hook，成功后 `return None` 吞掉 QQ/微信等非飞书平台的响应。

**验证**：在独立 Python 进程中用 Hermes 的 `Platform` Enum（20 个成员）逐一验证 `source.platform.value == "feishu"`，结果：
- 仅 `Platform.FEISHU` 返回 `True`
- `Platform.WEIXIN ("weixin")`、`Platform.WECOM ("wecom")`、`Platform.QQBOT ("qqbot")` 均不匹配
- `source.platform is None` 时抛 `AttributeError`，被外层 `except Exception: pass` 兜底

**改动**：`hermes_feishu_card/install/patcher.py`

三个 complete hook 渲染函数各改一行：

```python
# 改前：
if _hfc_card_delivered:
    return None

# 改后：
if _hfc_card_delivered and source.platform.value == "feishu":
    return None
```

影响函数：
- `_render_complete_hook_block()`（第 650 行）
- `_render_legacy_complete_hook_block()`（需要同步修改）
- `_render_previous_async_complete_hook_block()`（第 702 行）

### 3.2 #18a — DeepSeek 思维链标签支持

**背景**：DeepSeek API 官方通过独立的 `reasoning_content` 字段返回思考内容，但部分推理框架兼容层会将其嵌入 `content` 流并用 `` / `` 包裹。当前 `text.py` 只过滤 `<think>` / `</think>`。

**改动**：`hermes_feishu_card/text.py`

```python
# 第 7 行改前：
THINK_TAG_RE = re.compile(r"</?think>", re.IGNORECASE)
THINK_TAGS = ("<think>", "</think>")

# 改后：
THINK_TAG_RE = re.compile(r"</?think>|</?thinking>", re.IGNORECASE)
THINK_TAGS = ("<think>", "</think>", "<thinking>", "</thinking>")
```

`StreamingTextNormalizer._split_safe_text()` 已实现跨块前缀匹配，无需额外修改。

### 3.3 #18b — 工具次数正确统计

**根因**：`CardSession.apply()` 对 `tool.updated` 事件用 `tool_id` 做 key 覆盖写入 `self.tools`，`len(self.tools)` 返回的是去重后的工具种类数，而非实际调用次数（同一工具多次调用时计数不准）。

**改动**：`hermes_feishu_card/session.py`

```python
# CardSession 新增字段：
_tool_call_count: int = 0

# apply() 中 tool.updated 分支尾部增加：
self._tool_call_count += 1

# tool_count 属性改为：
@property
def tool_count(self) -> int:
    return self._tool_call_count

# tools 字典仍然保留（用于 _render_tool_summary 展示工具详情）
```

`render.py` 的 `_render_tool_summary()` 已使用 `session.tool_count` 和 `session.tools`，无需改动。

### 3.4 #18c — footer 旋转动画

**原理**：飞书 CardKit 的 Markdown 元素在流式模式下每次 PATCH 重渲染。利用 `time.time()` 驱动帧切换，无需额外 API 调用。

**改动**：`hermes_feishu_card/render.py`

```python
_SPINNER_FRAMES = ("⠋", "⠙", "⠹", "⠸", "⠼", "⠴", "⠦", "⠧", "⠇", "⠏")

def _spinner_text(label: str = "生成中") -> str:
    frame = _SPINNER_FRAMES[int(time.time() * 5) % len(_SPINNER_FRAMES)]
    return f"{frame} {label}"

# _render_footer() 中：
if session.status != "completed":
    return _spinner_text("生成中")
```

### 3.5 #10 — 卡片表格超限截断

**飞书限制**：每张卡片最多 5 个 Markdown `<table>`。超出报 `ErrCode: 11310`。

**改动 1**：`hermes_feishu_card/text.py` 新增检测函数：

```python
def count_markdown_tables(text: str) -> int:
    return len(re.findall(r'^\|[-: ]+\|', text, re.MULTILINE))

MAX_CARD_TABLES = 5
```

**改动 2**：`hermes_feishu_card/render.py` 的 `_render_main_content_elements()` 前加入截断逻辑：

```python
from .text import count_markdown_tables, MAX_CARD_TABLES

def _render_main_content_elements(main_text: str) -> list:
    # 表格超限截断
    table_count = count_markdown_tables(main_text)
    if table_count > MAX_CARD_TABLES:
        matches = list(re.finditer(r'^\|[-: ]+\|', main_text, re.MULTILINE))
        cutoff = matches[MAX_CARD_TABLES - 1].end()
        rest = main_text[cutoff:]
        next_para = re.search(r'\n\n', rest)
        if next_para:
            cutoff += next_para.start()
        main_text = main_text[:cutoff].rstrip() + (
            "\n\n> ⚠️ 内容含超过 5 个表格，超出部分已省略。"
        )
    
    chunks = _split_text(main_text, MAIN_CONTENT_CHUNK_CHARS)
    elements = []
    for index, chunk in enumerate(chunks):
        element_id = "main_content" if index == 0 else f"main_content_{index}"
        elements.append({"tag": "markdown", "element_id": element_id, "content": chunk})
    return elements
```

### 3.6 #16 — 进程内多 Profile 支持

**目标**：一个 sidecar 进程同时服务多个 Hermes profile，每个 profile 有独立的 feishu 凭据和会话隔离。

#### 3.6.1 配置模型

向后兼容：不配置 `profiles` 时行为与 V3.2.1 完全一致。

**新增 `profiles` schema**：

```yaml
# ── 单 profile（向后兼容，不变）──
feishu:
  app_id: "cli_xxx"
  app_secret: "..."

# ── 多 profile ──
profiles:
  default:                    # profile_id
    feishu:
      app_id: "cli_default_xxx"
      app_secret: "..."
    bots:
      default: default
      items: {}
    bindings:
      chats: {}
      fallback_bot: default

  work:                       # 另一个 profile_id
    feishu:
      app_id: "cli_work_yyy"
      app_secret: "..."
    bots:
      default: default
      items: {}
    bindings:
      chats: {}
      fallback_bot: default
```

**配置合并规则**：
1. 如果 `profiles` 存在且非空 → 使用多 profile 模式，忽略顶层 `feishu`/`bots`/`bindings`
2. 如果 `profiles` 不存在或为空 → 单 profile 模式，行为不变
3. 每个 profile 内的 `feishu`/`bots`/`bindings` 子配置遵循与顶层相同的 schema 和默认值

#### 3.6.2 Session 隔离

`server.py` 中 session key 从 `message_id` 扩展为复合 key：

- 单 profile：`key = message_id`（不变）
- 多 profile：`key = f"{profile_id}:{message_id}"`

`profile_id` 来源：`message.started` 事件的 `data.profile_id` 字段（V3.2.1 中 `hook_runtime._event_data()` 已从 `locals()` 提取并透传，但此前未使用）。若缺失则回退到 `"default"`。

#### 3.6.3 路由隔离

每个 profile 持有独立的 `FeishuBoundary`（包含独立的 `FeishuClientFactory` 和 `BotRegistry`）。

`runner.py` 中：
- 单 profile：创建单个 `FeishuBoundary`，传入 `create_app()`
- 多 profile：创建 `dict[profile_id, FeishuBoundary]`

`server.py` 中 `_resolve_route()` / `_client_for_bot()` 根据事件的 `profile_id` 选择对应 boundary。

#### 3.6.4 健康检查

`/health` 响应新增 `profiles` 字段，按 profile 分组展示：

```json
{
  "status": "healthy",
  "active_sessions": 3,
  "profiles": {
    "default": {"active_sessions": 2, "metrics": {...}},
    "work": {"active_sessions": 1, "metrics": {...}}
  },
  "sessions": {...}
}
```

单 profile 模式下不返回 `profiles` 字段（保持兼容）。

#### 3.6.5 用户配置指南

**单 profile 用户（无需改动）**：现有 `feishu.app_id` / `feishu.app_secret` 配置继续生效。

**多 profile 用户配置步骤**：

```bash
# 1. 编辑配置文件 ~/.hermes_feishu_card/config.yaml
# 将原来的 feishu 块替换为 profiles 块

# 2. 重启 sidecar
hermes-feishu-card restart

# 3. 验证
curl http://127.0.0.1:8765/health | jq '.profiles'
```

**Hermes 侧无需任何配置**：`hook_runtime._event_data()` 在 `message.started` 事件中自动提取 `profile_id` 并透传。
